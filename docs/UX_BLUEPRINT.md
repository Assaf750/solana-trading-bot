# UX_BLUEPRINT — Soltrade Terminal — إعادة الهيكلة على معيار المنصّات الكبرى

> مخطّط منهجي مدروس لتجربة المستخدم والواجهة. المعيار: **أيّ تجربة أقل من GMGN/Axiom/Photon/Trojan مرفوضة.** هذا المستند يقود إعادة بناء الصفحات. ليس وثيقة SSOT (`docs/00`–`12` مرجع الحوكمة).

**التاريخ:** 2026-06-12 · **النطاق:** الواجهة (`apps/operator-ui`) فوق الخادم القائم (read-only/fail-closed محفوظ).

---

## 1. ما الذي يجعل المنصّات الكبرى «كبرى» (المبادئ الملزِمة)

1. **الكثافة الذكية:** أقصى معلومات مفيدة لكل شاشة، مرتّبة بصرياً (لا فراغ، لا بطاقات هزيلة). أرقام monospace ملوّنة دلالياً.
2. **مسار عمل لا صفحات منفصلة:** المستخدم لا «يتنقّل بين تبويبات» بل ينجز مهمّة متّصلة: اكتشف → صفِّ → افحص → انسخ → راقب — كل ذلك بأقل نقرات.
3. **لوحة قائمة + تفصيل (master–detail):** جدول كثيف على اليسار، تفاصيل المختار على اليمين، دون مغادرة الصفحة.
4. **تصفية وفرز فوريان:** أشرطة فلاتر، أعمدة قابلة للفرز، بحث لحظي.
5. **حالة حيّة دائمة:** نبض الطرفية (أسعار، slot، latency، حالة المحرّك) ظاهر باستمرار.
6. **إجراءات سريعة (quick actions):** زرّ واحد للنسخ/المتابعة/الإيقاف من أي صف.
7. **صدق البيانات:** ما لا يتوفّر يُعرض «unavailable» لا صفراً — وهذا ميزة ثقة لا عيب.

## 2. المهام الجوهرية الخمس (Core Jobs) — التطبيق يدور حولها

| # | المهمة | الصفحة/الواجهة | الحالة |
|---|---|---|---|
| J1 | **اكتشاف** محافظ مرشّحة للنسخ | Wallet Discovery (leaderboard) | يحتاج مصدر بيانات (provider/scan) — Phase 3 |
| J2 | **تصفية وفرز** المحافظ حسب الأداء/الخطر | Wallet Workspace (filter+sort) | **Phase 1 — الآن** |
| J3 | **فحص** محفظة بعمق قبل النسخ | Detail/Analysis panel | جاهز (المحلّل) — يُدمج في Workspace |
| J4 | **نسخ/متابعة** بإعدادات واضحة | Copy controls داخل الـ detail | جاهز — يُعاد تصميمه |
| J5 | **استعراض/مراقبة** النسخ الحيّ والأداء | Trading Workspace + Analytics | Phase 2 |

> J1 (اكتشاف محافظ جديدة من leaderboard عام) يتطلّب **مصدر بيانات**: إمّا مزوّد (مثل بيانات GMGN/Birdeye API) أو فهرسة on-chain للمحافظ النشطة. لا يُختلق leaderboard. يُحسم مصدره ثم يُبنى (Phase 3).

## 3. نظام التخطيط (Layout System) — يُضاف لمكتبة المكوّنات

- **`.workspace`**: شبكة master–detail (قائمة مرنة + لوحة تفصيل لاصقة) تنهار لعمود واحد على الشاشات الصغيرة.
- **`.filterbar`**: شريط فلاتر أفقي (بحث + شرائح فلترة + فرز) فوق القائمة.
- **`.wrow`**: صف محفظة كثيف قابل للنقر/الاختيار مع إجراءات سريعة.
- **`.detail-pane`**: لوحة تفصيل لاصقة (sticky) للعنصر المختار.
- **`.stattile`**: بلاطة إحصاء موحّدة (label + رقم ملوّن دلالياً).
- **`.kpi-strip`**: شريط مؤشّرات علوي لكل صفحة (ملخّص فوري).

كلها أنماط عرض بحتة (Category A)، تُعيد استخدام التوكنات القائمة، ولا تمسّ الخلفية.

## 4. إعادة هيكلة الصفحات (الترتيب التنفيذي)

### Phase 1 — Wallet Workspace (J2+J3+J4) ← **يُنفَّذ الآن**
يحوّل صفحة «ذكاء المحافظ» من نموذج تسجيل بدائي إلى **كونسول بحث/تصفية/نسخ** كامل:
- **kpi-strip**: عدد المحافظ · المتابَعة · متوسط win rate (للمحلَّلة) · المحرّك.
- **filterbar**: بحث بالعنوان/الاسم · فلترة (الكل/متابَعة/متوقفة/محلَّلة) · فرز (win rate · realized PnL · trades · الأحدث).
- **قائمة كثيفة** (`.wrow`): عنوان/اسم · copy_mode · متابعة (toggle) · win rate · realized PnL · trades · علم بوت — كلها أعمدة، إجراءات سريعة (تحليل/نسخ/حذف). النقر يفتح التفصيل.
- **detail-pane** لاصقة: تحليل المحفظة الكامل (المحلّل القائم) + إعدادات النسخ (copy_mode · TP/SL · slippage) + زرّ متابعة بارز.
- **«تحليل الكل»**: يحلّل كل المحافظ المتابَعة دفعةً (محكوم بمعدّل).

### Phase 2 — Trading Workspace + Command Center (J5)
- Command Center → لوحة قيادة بشريط KPI + بطاقات حيّة كثيفة (mini-charts، آخر قرارات، صحّة المزوّد).
- Trading Workspace → master–detail للمراكز: قائمة مراكز كثيفة + لوحة تفصيل (Decision Trace + P&L + خروج). جدول صفقات حيّ ملوّن.

### Phase 3 — Wallet Discovery (J1) + Analytics
- Discovery: leaderboard محافظ قابل للبحث/الفرز (بعد حسم مصدر البيانات) → «أضف للمتابعة» بنقرة.
- Analytics: تقارير كثيفة (calendar PnL، توزيعات، مقارنات) على نمط GMGN.

### Phase 4 — صقل شامل
حالات تحميل/فراغ/خطأ مصقولة لكل صفحة · اختصارات لوحة مفاتيح · تجاوب RTL كامل · حركات دقيقة.

## 5. حدود ثابتة أثناء إعادة الهيكلة
- تصميم/عرض فقط (Category A) — لا تغيير سلوك تداول/تنفيذ، لا مسّ خلفية أو حوكمة.
- صدق البيانات: «unavailable» لا اختلاق · simulated/real لا يُخلطان.
- المجموعة 1899/1899 والحارسان يبقيان أخضرين بعد كل مرحلة.
- كل مرحلة: بناء + لقطة تحقّق بصرية + commit + push.
